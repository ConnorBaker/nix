# Eval-Trace Rearchitecture Proposal (single commit, no shims)

Rearchitect eval-trace into **TraceStorage** (abstract base, two concrete
backends) + **Verifier** + **Recorder** + **TraceBackend**. Delete
`TraceStore`, the file-static `VerifyImpl`, `VerificationOrchestrator`,
and `AsyncRuntime`.

Delivery: single commit, no transitional aliases. No schema change, no
blob-format change, no `kSchemaEpoch` bump — old DB files continue to
work.

Performance claims below cite `eval-trace-bench-results/{hot,cold,reference}/0`,
generated by `nix run .#eval-trace-bench -- generate --num-commits 100
--run-number 0` (workload `nixos/release.nix closures.gnome` evaluated
against 100 adjacent nixpkgs commits; see Appendix B for aggregates).

## 1. Target layering

### 1.1 Before (current)

```
EvalState
  └── TraceRuntime
        └── makeTraceBackend() → StoreTraceBackend
                                   ├── shared_ptr<TraceStore>       ← god class:
                                   │     SQLite + 28 stmts + mutex, dedup maps,
                                   │     session caches, pending vectors, ID
                                   │     counters, SetOnce<SessionConfig>,
                                   │     zombie VerificationSession (never read)
                                   └── unique_ptr<AsyncRuntime>
                                         ├── io_context + scheduler + pools
                                         └── unique_ptr<VerificationOrchestrator>
                                               ├── VerificationSession (the live one)
                                               ├── PrefetchPool (StrandLocal)
                                               └── verifyAttrImpl (awaitable)

trace-store-verify.cc  (file-static) — OriginScopeFactory, FileStrandGate,
     StorePathBatch, VerifyContext, VerificationState, Pass2Result, VerifyImpl
     (~25 static methods), RecoveryState<Stage>.
```

Baseline sizes, verified: 1,136 lines of `trace-store.hh`, 835 of
`trace-store.cc`, 773 of `trace-store-lifecycle.cc`, 1,912 of
`trace-store-verify.cc`, 220 of `verification-orchestrator.cc`, 138 of
`async-runtime.hh`. The `struct State` inside `TraceStore` holds **28
SQLiteStmts**.

### 1.2 After

```
EvalState
  └── TraceRuntime                           (unchanged)
        └── makeTraceBackend() → unique_ptr<TraceBackend>

TraceBackend                                 (replaces AsyncRuntime + StoreTraceBackend)
  ├── unique_ptr<TraceStorage>               — abstract; polymorphic backend
  ├── io_context + scheduler + pools         — ex-AsyncRuntime members
  ├── unique_ptr<Verifier>                   — session-scoped verification
  ├── unique_ptr<Recorder>                   — one-shot record driver
  └── 10 public methods + friend-access from TraceSession             — §2.4

TraceStorage  (abstract base)                — pure virtuals; §2.1
  ├── SqliteTraceStorage                     — today's TraceStore persistence
  └── InMemoryTraceRows                      — in-process trace rows; AttrVocabStore
                                               still persists via TraceRuntime

Verifier  (merges VerificationOrchestrator + VerifyImpl)  — §2.2
  ├── VerificationSession (value member)           — L1 cache, subsumption,
  │                                                  cycle-detection, ParseCaches
  ├── StrandLocal<PrefetchPool,                    — gated by Certifier<
  │               VerificationAccessTag>             VerificationAccessTag>
  ├── six session-scoped read-through caches       — traceHeaderCache,
  │                                                  traceFullCache,
  │                                                  currentNodeIndex,
  │                                                  resultPayloadCache,
  │                                                  depKeySetCache,
  │                                                  deferredTraceBlobs
  └── 2 public methods + friend-access from TraceBackend + tests

Recorder  (one-shot BSàlC recorder)          — §2.3
  └── 1 public method: record(ea, pathId, value, deps, observer?)

TraceSession  (surface unchanged)
  └── unique_ptr<TraceBackend> runtime_
```

`trace-store-verify.cc`'s file-static helpers (`OriginScopeFactory`,
`FileStrandGate`, `StorePathBatch`, `VerificationState`, `Pass2Result`,
`VerifyImpl`, `RecoveryState<Stage>`) move into `verifier.cc` as
file-static or `Verifier`'s private helpers. The `VerifyContext`
parameter bundle is deleted entirely: after the migration, Verifier's
private methods take `ea` directly and reach `pools`/`vocab`/`symbols`/
`registry`/`state`/`session` as members.

## 2. Interfaces

### 2.1 `TraceStorage` (abstract base)

```cpp
namespace nix::eval_trace {

// ── Capability token ────────────────────────────────────────────────

class ExclusiveTraceStorageAccess {
    TraceStorage & storage_;
    const gdp::Proof<BlockingTag> & bs_;
    ExclusiveTraceStorageAccess(TraceStorage & s, const gdp::Proof<BlockingTag> & bs)
        : storage_(s), bs_(bs) {}
    friend class TraceStorage;

public:
    ExclusiveTraceStorageAccess(const ExclusiveTraceStorageAccess &) = delete;
    ExclusiveTraceStorageAccess(ExclusiveTraceStorageAccess &&) = delete;

    /// `ea` implies `bs`: the mutex-holding thread is also the blocking
    /// thread. Callers pass this to non-storage blocking ops (filesystem,
    /// daemon IPC, git). MUST NOT be used to call
    /// `storage.withExclusiveAccess(bs, ...)` on the same storage.
    const gdp::Proof<BlockingTag> & blockingProof() const noexcept { return bs_; }
};

// ── Namespace-scope value types ──────────────────────────────────────

// Split into two types because NodeStamp is a Sessions-table concept;
// History rows have no node_stamp column (schema at
// trace-store-lifecycle.cc:229-236). The zero-nodeStamp sentinel
// previously meant two different things (legacy pre-stamp row vs.
// history row). Split makes the invariant structural.
struct CurrentNodeRef { TraceId traceId; ResultId resultId; NodeStamp nodeStamp; };
struct HistoryNodeRef { TraceId traceId; ResultId resultId; };

struct HistoryEntry   { DepKeySetId depKeySetId; TraceId traceId; ResultId resultId; TraceHash traceHash; };
struct TraceHeader    { TraceHash traceHash; DepKeySetHash keySetHash; DepKeySetId depKeySetId; };
struct ResultPayload  { ResultKind type; uint32_t encodingVersion; std::string payload; std::string auxContext; };

// Blobs are non-optional — schema guarantees both columns NOT NULL (may
// be empty). See trace-store-lifecycle.cc:201, :213.
struct TraceBlobs     { TraceHeader header; std::vector<uint8_t> keysBlob; std::vector<uint8_t> valuesBlob; };

struct VerifyResult   { CachedResult value; TraceId traceId; };
struct RecordResult   { TraceId traceId; };
struct ResolvedDep    { /* promoted from nested TraceStore::ResolvedDep */ };

struct RuntimeRootRecord {
    DepSource source;
    RuntimeFetchIdentityDepKey fetchIdentity;
    RuntimeRootNarHash narHash;
    RuntimeRootStorePath storePath;
};
struct VerifiedRuntimeRootRecord { RuntimeRootRecord record; SourcePath rootPath; };
struct RuntimeRootLoadResult     { size_t storedCount{}, rejectedCount{}; std::vector<RuntimeRootRecord> entries; };

struct EvalInfoRecord {
    enum class Source { Session, History };
    TraceId traceId;
    ResultId resultId;
    TraceHash traceHash;
    DepKeySetHash keySetHash;
    DepKeySetId depKeySetId;
    CachedResult value;
    std::shared_ptr<const std::vector<Dep>> deps;
    Source source;
};

// ── Abstract base ────────────────────────────────────────────────────

class TraceStorage {
    /// Combined-mutex invariant (§6): storeMutex_ serializes ALL mutable
    /// state — storage's SQLite connection, dedup maps, pending buffers,
    /// ID counters, AND verifier-owned state (session-scoped caches,
    /// VerificationSession, ParseCaches). This is why FileStrandGate
    /// derives its proof from `ea` alone.
    ///
    /// Non-recursive. Every private helper takes `ea`, so callers never
    /// re-enter withExclusiveAccess. A thread_local re-entrancy detector
    /// keyed on `TraceStorage *` catches accidental nesting: debug builds
    /// assert; release builds deadlock (strictly better than today's
    /// silent recursion on recursive_mutex, which masks the bug).
    mutable std::mutex storeMutex_;

    std::optional<SessionConfig> sessionConfig_;
    SemanticSessionKey semanticSessionKey_;
    SessionRecoveryKey stableRecoveryKey_;
    std::atomic<uint32_t> nextNodeStamp_{1};

    void withExclusiveAccessReentrancyCheckEnter();
    void withExclusiveAccessReentrancyCheckExit() noexcept;

protected:
    explicit TraceStorage(SemanticSessionKey bootstrapKey);
    virtual ~TraceStorage() = default;

    void raiseNextNodeStamp(uint32_t floor) noexcept;

public:
    TraceStorage(const TraceStorage &) = delete;
    TraceStorage & operator=(const TraceStorage &) = delete;

    template<typename F>
    auto withExclusiveAccess(const gdp::Proof<BlockingTag> & bs, F && f)
        -> std::invoke_result_t<F, const ExclusiveTraceStorageAccess &>;

    /// SetOnce semantic: throws on second call.
    void setSessionConfig(SessionConfig config);
    const SemanticSessionKey & currentSemanticSessionKey() const noexcept { return semanticSessionKey_; }
    const SessionRecoveryKey & currentStableRecoveryKey()  const noexcept { return stableRecoveryKey_; }
    NodeStamp allocateNodeStamp() noexcept
    { return NodeStamp(nextNodeStamp_.fetch_add(1)); }

    // ── Content-addressed writes. Payloads taken by rvalue reference:
    //     moves into backend-owned storage instead of copying zstd-compressed blobs.
    virtual ResultId     insertResult   (const ExclusiveTraceStorageAccess &,
                                         ResultHash, EncodedResultPayload &&) = 0;
    virtual DepKeySetId  insertDepKeySet(const ExclusiveTraceStorageAccess &,
                                         DepKeySetHash, std::vector<uint8_t> && keysBlob) = 0;
    virtual TraceId      insertTrace    (const ExclusiveTraceStorageAccess &,
                                         FullTraceHash, TraceHeader,
                                         std::vector<uint8_t> && valuesBlob) = 0;

    // ── Reads. Two trace-load virtuals: lightweight header vs. full blobs.
    //
    // Benchmark justification: the hot workload shows avg trace blob size
    // of 661 KB (49 KB keys + 612 KB values) across 93 traces. Today's
    // ensureTraceHeader pulls the full 661 KB just to extract 72 bytes
    // of scalar header — a 9,000× I/O amplification. Splitting the
    // virtuals lets header-only callers (resolveTraceContextHash,
    // tryStructuralVariantRecovery, getCurrentTraceHash, queryEvalInfo)
    // avoid BLOB reads on cache miss.
    virtual std::optional<TraceHeader> loadTraceHeader (const ExclusiveTraceStorageAccess &, TraceId) = 0;
    virtual std::optional<TraceBlobs>  loadTraceBlobs  (const ExclusiveTraceStorageAccess &, TraceId) = 0;
    virtual std::optional<std::vector<uint8_t>> loadKeysBlob     (const ExclusiveTraceStorageAccess &, DepKeySetId) = 0;
    virtual std::optional<ResultPayload>        loadResultPayload(const ExclusiveTraceStorageAccess &, ResultId) = 0;

    // Current/history lookup. Split types: a CurrentNodeRef always carries
    // a stamp; a HistoryNodeRef never does.
    virtual std::optional<CurrentNodeRef> lookupCurrent      (const ExclusiveTraceStorageAccess &, AttrPathId) = 0;
    virtual std::optional<HistoryNodeRef> lookupLatestHistory(const ExclusiveTraceStorageAccess &, AttrPathId) = 0;

    // Three named history-scan virtuals (not a HistoryFilter variant):
    // each maps 1:1 to a distinct SQLite prepared statement. Cardinality
    // is fixed at three; variant dispatch adds overhead for no gain.
    virtual std::vector<HistoryEntry> queryAllHistory           (const ExclusiveTraceStorageAccess &, AttrPathId) = 0;
    virtual std::vector<HistoryEntry> queryHistoryByTraceHash   (const ExclusiveTraceStorageAccess &, AttrPathId, TraceHash) = 0;
    virtual std::vector<HistoryEntry> queryHistoryByGitIdentity (const ExclusiveTraceStorageAccess &, AttrPathId, CurrentGitIdentityHash) = 0;

    // Publish. Split by name, not by bool+correlated-optional: the
    // optional gitIdentityHash is only meaningful when inserting a
    // History row. Two named methods make the invariant structural.
    virtual CurrentNodeRef publishFreshRecord(
        const ExclusiveTraceStorageAccess &,
        AttrPathId, TraceId, ResultId,
        std::optional<EvalTraceHash> gitIdentityHash) = 0;
    virtual CurrentNodeRef publishHistoryBootstrap(
        const ExclusiveTraceStorageAccess &,
        AttrPathId, TraceId, ResultId) = 0;

    virtual void                  recordRuntimeRoot(const ExclusiveTraceStorageAccess &,
                                                    const RuntimeRootRecord &, Store &) = 0;
    virtual RuntimeRootLoadResult loadRuntimeRoots (const ExclusiveTraceStorageAccess &, Store &) = 0;

    virtual void flush(const ExclusiveTraceStorageAccess &) = 0;
};

} // namespace nix::eval_trace
```

**What's dropped and why (verified).**

- **`attrExists`** — ten test callers; today's body is literally
  `lookupCurrentNode(bs, pathId).has_value()` (trace-store.cc:611-614).
  Callers switch to `storage.lookupCurrent(ea, pathId).has_value()`.
- **`storage()` accessor on `ExclusiveTraceStorageAccess`** — callers
  already hold a storage reference; round-tripping through ea is a wart.
- **`HistoryFilter` variant** — three distinct prepared statements;
  variant buys nothing.
- **`insertHistory` bool + correlated `optional<gitIdentityHash>`** —
  the optional is only valid when the bool is true; that's an invariant,
  not an API.
- **`resultPayloadCache` on storage** — verified via benchmark: the
  whole hot run (100 commits, 700 verify calls) has 38 `Results` rows
  totalling 4,335 bytes, mean payload 114 B. Every verify decodes its
  result once; there is zero within-session reuse in hot mode (since
  eval-trace-bench runs each commit as a subprocess). Cache moves to
  Verifier where it's session-scoped. Storage has no result cache.

**What's added and why.**

- **`loadTraceHeader`** — new virtual, header-only read. Motivated by
  benchmark blob sizes (avg 661 KB/trace); today's `ensureTraceHeader`
  reads the full blob to extract 72 B.
- **`CurrentNodeRef` / `HistoryNodeRef` split** — makes "History rows
  have no NodeStamp" structural.
- **`TraceBlobs.keysBlob`/`valuesBlob` non-optional** — schema NOT NULL
  guarantees they always exist (may be empty).

**Design notes.**

- **Base-owned state (SessionConfig, session keys, nodeStamp counter)**
  because both backends share identical semantics. Only backend-specific
  work is virtual.
- **Raw blobs, not deserialized Dep vectors.** Deserialization +
  session-scoped caching lives on `Verifier`, keeping the storage
  interface agnostic to the Dep/Key/Result object model.
- **Single `flush(ea)` virtual, no `flushExclusive`.** Today's
  `TraceStore` has both. `flushExclusive()` exists because
  `StoreTraceBackend::flush()` and `TraceStore::~TraceStore()` need to
  flush without an `ea` available. In the new design, the one non-dtor
  caller (`TraceBackend::flush`) self-mints `bs` and acquires `ea` via
  `withExclusiveAccess`, then calls `flush(ea)`. Concrete backends still
  do their own dtor commit privately (not via a virtual).
- **Abstract base, not templates.** Templating `Verifier<S>` and
  `Recorder<S>` on a concrete storage type would propagate the template
  parameter up the ownership chain: `TraceBackend<S>` (holds
  `unique_ptr<Verifier<S>>`), then `TraceSession<S>`, then `TracedExpr<S>`,
  then `Value<S>`. `Value` is a 24-byte union shared across the
  evaluator — templating it is a non-starter. Virtual dispatch is
  outside the SV inner loop (storage boundary is crossed only on the
  cold path of each trace, not each dep), so the runtime cost is not
  meaningful. Benchmark confirms: loadTrace is 1.2% of verify time;
  Pass 1/2 classifies ~24,400 deps per verify (17.1M total / 700
  verifies, 100 commits) and ~12,257 of those reach the dep-hash L1
  path, all with no storage virtuals.

**Design notes.**

- **Public surface is 2 entry points.**
  `VerificationOrchestrator::submitPrefetchHint` (singular) is deleted —
  verified zero external callers (only its own declaration at
  verification-orchestrator.hh:88 and definition at :74). Session
  lifecycle (`bindSession`, `resetVerificationState`) and the
  `getCurrentTraceHashAsync` awaitable bridge are private because
  `TraceBackend` is their only caller.
- **`VerifyContext` parameter bundle is deleted.** Today's struct
  (trace-store-verify.cc:434-442) bundles `{ea, bs, store, pools,
  registry, state, session}`. After migration: `bs = ea.blockingProof()`;
  `store → storage_` member; `pools/vocab → VocabAwareHasher` base;
  `registry/state/session` are members set by `bindSession()` +
  construction. Only `ea` is per-call.
- **Private helpers take `ea`, not `bs`.** Every private helper on
  Verifier, SqliteTraceStorage, and the abstract TraceStorage takes
  `const ExclusiveTraceStorageAccess &`. Enumerated against the code:
  the three current mutex-acquiring private helpers (`ensureTraceHeader`,
  `lookupCurrentNode`, `loadResultPayload` at trace-store.cc:192, :552,
  :588) are each reached only from inside a `withExclusiveAccess` scope.
  Taking `ea` promotes this invariant from convention to type.
- **`_ForTest` entries for the synchronous test path.** ~252 call sites
  across 30 test files drive verify/recovery directly against a storage
  handle without an AsyncRuntime. These entries take an already-acquired
  `ea` and share their body with the async path via private `*Impl`
  helpers.
- **`TraceObserver` inheritance wires cache coherence** (§3).
  `Recorder::record` fires the observer callbacks inside its
  storage-lock scope; Verifier's overrides update the session-scoped
  read-through caches while `ea` is still held.
- **`lookupTraceHeader`** wraps `traceHeaderCache_` + falls through to
  `storage_->loadTraceHeader(ea, …)`. Avoids the 661 KB → 72 B I/O
  amplification documented in §2.1.

### 2.3 `Recorder`

```cpp
namespace nix::eval_trace {

/// Interface Verifier implements to receive new-trace / publish-current
/// callbacks from Recorder, fired inside the storage-lock scope so the
/// verifier's session-scoped caches update atomically with the DB write.
struct TraceObserver {
    virtual ~TraceObserver() = default;
    virtual void onNewTrace(
        TraceId, const TraceHeader &,
        const std::vector<Dep> & fullDeps,
        DepKeySetId, const std::vector<Dep::Key> & keys) = 0;
    virtual void onPublishCurrent(AttrPathId, const CurrentNodeRef &) = 0;
};

/// One-shot BSàlC constructive-trace recorder. Stateless beyond refs
/// established at construction; the class shape caps argument-list
/// growth and makes the vocab-aware-hashing concern (via the private
/// VocabAwareHasher base) explicit.
class Recorder : private VocabAwareHasher {
public:
    Recorder(TraceStorage & storage, InterningPools & pools, AttrVocabStore & vocab) noexcept
        : VocabAwareHasher(pools, vocab), storage_(storage) {}

    Recorder(const Recorder &) = delete;
    Recorder & operator=(const Recorder &) = delete;

    /// - `ea` proves exclusive storage access.
    /// - `pathId` must be live in `vocab_`.
    /// - `observer`, if non-null, receives `onNewTrace` + `onPublishCurrent`
    ///   callbacks before `ea` is released.
    RecordResult record(const ExclusiveTraceStorageAccess & ea,
                        AttrPathId pathId,
                        const CachedResult & value,
                        const std::vector<Dep> & allDeps,
                        TraceObserver * observer = nullptr);

private:
    TraceStorage & storage_;
    // pools_ + vocab_ inherited from VocabAwareHasher.
    // NO SymbolTable: verified that encodeCachedResult
    // (trace-result-codec.cc:428-481) does not reference SymbolTable.
    // Only decode needs symbols (line 495), and decode lives on Verifier
    // / on the decode free function.
};

} // namespace nix::eval_trace
```

**Ownership.** `TraceBackend` owns one `Recorder` member alongside its
`Verifier`. Tests that record directly against a raw storage construct a
local `Recorder`.

**`record(...)` body:**

1. `sortAndDedupDeps(allDeps)`.
2. `computeTraceHashFromSorted / FullTraceHashFromSorted /
   DepKeySetHashFromSorted` via `VocabAwareHasher::feedKey`.
3. `serializeKeys(sortedKeys)` / `serializeValues(sorted)` — free fns in
   `store/trace-serialize.hh`.
4. `encodeCachedResult(value, vocab_)` — free fn taking
   `(CachedResult, AttrVocabStore &)`.
5. `storage_.insertResult / insertDepKeySet / insertTrace` (payloads
   moved via rvalue-ref).
6. `extractGitIdentityHash(sortedDeps)` — free fn in `deps/analysis.hh`.
7. `auto ref = storage_.publishFreshRecord(ea, pathId, traceId, resultId,
   gitIdHash);` — storage allocates NodeStamp internally, returns
   `CurrentNodeRef`.
8. If `observer`, fire `onNewTrace(traceId, header, sortedDeps, dksId,
   keys)` and `onPublishCurrent(pathId, ref)` while `ea` is still held.

**Benchmark note.** Cold mode records 0.93 traces per commit (93 total
across 100 commits — content-addressed dedup collapses `2 workload
outputs × 100 commits = 200` into 93 unique full_hash rows). Hot mode:
zero records. Recorder's single-method surface is right-sized for this
volume.

### 2.3.1 `VocabAwareHasher` mixin

`feedKey` is a dep-key hasher that closes over `InterningPools` and
`AttrVocabStore`. Both Recorder and Verifier need it; neither wants to
own the refs as separate member fields. The mixin binds them once via
private inheritance.

```cpp
namespace nix::eval_trace {

/// EBO-safe mixin supplying vocab-aware dep-key hashing. Concrete users
/// (Recorder, Verifier) privately inherit and get `feedKey` as a
/// protected inline member. Zero-cost under EBO.
class VocabAwareHasher {
    InterningPools & pools_;
    AttrVocabStore & vocab_;
protected:
    VocabAwareHasher(InterningPools & pools, AttrVocabStore & vocab) noexcept
        : pools_(pools), vocab_(vocab) {}

    void feedKey(CanonicalHashBuilder & builder, const Dep::Key & key) const {
        if (key.isTraceContext()) {
            builder.field("dep.key.kind", key.kind);
            vocab_.feedPath(builder, key.attrPathId);
            return;
        }
        feedCanonicalDepKeyMaterial(builder, pools_, key);
    }
};

} // namespace nix::eval_trace
```

Header: `store/vocab-aware-hasher.hh` (~40 lines, header-only).
`AttrVocabStore` lives under `store/`, so the mixin header lives under
`store/` too. `hash.hh` cannot take this dependency without regressing
the include DAG.

  std::optional<EvalInfoRecord> TraceBackend::queryEvalInfo(
      AttrPathId pathId, bool allowHistoryFallback) {
      return gdp::Certifier<BlockingTag>::withProof([&](const auto & bs) {
          return storage_->withExclusiveAccess(bs, [&](const auto & ea)
                                                 -> std::optional<EvalInfoRecord> {

                      traceId = h->traceId; resultId = h->resultId;
                      source = EvalInfoRecord::Source::History;

                  .traceHash = header->traceHash,
                  .keySetHash = header->keySetHash,

                  .value = std::move(value),
                  .deps = std::move(deps),

              };
          });
      });

## 3. Publishing and cache coherence

`Recorder::record` holds `ea` across the `insertX → publishFreshRecord`
sequence. Verifier's session-scoped read-through caches must update
inside the same critical region so a concurrent reader never observes a
DB state newer than the cache.

**Mechanism.** `Recorder::record` takes a `TraceObserver *` parameter.
`TraceBackend::record` passes `&verifier_` (Verifier privately inherits
`TraceObserver`). Tests constructing a local `Recorder` against a raw
`TraceStorage` pass `nullptr`.

```cpp
void Verifier::onNewTrace(TraceId id, const TraceHeader & h,
                          const std::vector<Dep> & fullDeps,
                          DepKeySetId dksId,
                          const std::vector<Dep::Key> & keys) {
    traceHeaderCache_.insert_or_assign(id, h);
    traceFullCache_  .insert_or_assign(id, std::make_shared<const std::vector<Dep>>(fullDeps));
    depKeySetCache_  .insert_or_assign(dksId, std::make_shared<const std::vector<Dep::Key>>(keys));
}
void Verifier::onPublishCurrent(AttrPathId p, const CurrentNodeRef & r) {
    currentNodeIndex_.insert_or_assign(p, r);
}
```

**Verifier's own publish path (history-bootstrap success)** updates
`currentNodeIndex_` directly after calling
`storage_->publishHistoryBootstrap` — no observer round-trip, since
Verifier already owns the cache. The observer exists so Recorder
(consumed by tests without a Verifier) stays Verifier-agnostic.

**Alternatives rejected.** Caller-side "update after" requires a second
code path for tests that don't have a Verifier. Putting the observer on
`TraceStorage::publishFreshRecord` forces both backends to plumb the
pre-blob view into the observer — a parameter the storage interface
otherwise never needs. Recorder-side observer keeps the storage
interface lean.

**Lifetime.** The observer is dereferenced only during
`recorder.record(ea, ..., observer)`, while `ea` is held. In production
`verifier_` and `recorder_` are co-owned by `TraceBackend`, so verifier
always outlives any record call.

**Benchmark check.** Observer fires at most 93 times per cold run (per
record, one per recorded trace). Hot mode: zero observer fires. Cost is
negligible.

## 5. Preserved invariants

### 5.1 Soundness barriers

All type-level barriers move with their owners; none change shape:

- **Three-gate subsumption barrier.** `grantVerifiedSubsumption<O>`
  retains `requires std::same_as<O, dep_origin::CurrentTrace>`;
  `OriginDep<Origin>` keeps private per-specialization ctors;
  `VerifiedSubsumption` default ctor stays private with
  `grantVerifiedSubsumption` as sole friend. Types and their friend
  decls move from `trace-store.hh` to `verifier.hh`.
- **`OriginScopeFactory` ODR barrier.** Forward-declared in
  `verifier.hh`, defined only in `verifier.cc`. Spelling
  `enterCurrentTrace` from another TU fails linking.
- **CurrentTraceDep vs CandidateDep code-path selection.**
  `if constexpr (std::is_same_v<Origin, CurrentTrace>)` in
  `resolveDepHash` — the `static_assert` that forces any new origin
  to audit the branch stays.
- **`ExclusiveTraceStorageAccess`.** Minted only by
  `TraceStorage::withExclusiveAccess`. Holding `ea` implies the storage
  mutex is held for the callback's duration. Copy/move deleted.
- **`FileStrandTag` derived from `ea`.**
  `FileStrandGate::ifPassed(const ExclusiveTraceStorageAccess &, f)`
  mints the proof. `ParseCaches` lives on `VerificationSession` (owned
  by Verifier); acquiring `ea` on storage grants exclusive access to
  `ParseCaches` through the same mutex.
- **`BlockingTag` discipline.** Public abstract-base operations all
  take `ea` — no `bs`-only entry on `TraceStorage`.
  `SqliteTraceStorage` privately inherits `Certifier<BlockingTag>` for
  its own lifecycle paths (ctor `bulkLoadAllLocked`, dtor commit,
  internal flushes that predate `ea`). `InMemoryTraceRows` does no
  blocking I/O and does not inherit the certifier.
- **`VerificationAccessTag`.** Private-inherited by `Verifier`;
  `prefetchPool_` is `StrandLocal<PrefetchPool, VerificationAccessTag>`.
- **Trace-scoped subsumption.** `isFileVerified(traceId, key)` and
  `verifiedContentFilesByTrace_` stay private on `VerificationSession`.
- **Cycle detection.** `inProgressTraceIds` + RAII guard on
  `verifyTrace` entry/exit.
- **`DepResolution<State, TaggedDepType>` typestate.** L1→L3 ordering
  via `Linear<T>` CRTP. Subsumption is checked inside the
  `resolveDepHash` free fn.
- **`RecoveryState<Stage>` typestate.** Untried → GitMissed →
  DirectMissed, `Linear<T>` CRTP.
- **Pass-2 `ValidViaImplicitShapeOverride` trace-hash patch.** Writes
  only to the session-scoped `traceHeaderCache_`; no DB write
  (persisting the patched hash would break the content-address
  invariant of `Traces.trace_hash`).
- **Schema.** Tables, column shapes, indexes, pragmas, blob formats
  identical. Filename stays `eval-trace-v25-<algo>.sqlite`.

### 5.2 Hot-path performance preserved

Benchmark measurements from `eval-trace-bench-results/hot/0` (100
commits, closures.gnome.{x86_64,aarch64}-linux):

- verify = **43% of wall time** (42.5s / 99.5s total)
- verifyTrace = **98% of verify time** (where depHash work dominates)
- loadTrace = **1.2% of verify time** (506 ms / 42.5s) and **0.5% of
  wall time** — storage I/O is not the bottleneck
- Storage calls per verify ≈ 2–3 (lookupCurrent + loadTraceHeader/
  loadTraceBlobs + loadResultPayload)

Virtual dispatch crosses the storage boundary at most a handful of
times per verify; the inner Pass 1/2 classification loop runs ~24,400
iterations per verify (17.1M `verify.depsChecked` / 700 verifies across
100 commits), and ~12,257 of those per verify reach
`Verifier::resolveCurrentDepHash` (8.58M `depHash.cacheHits +
cacheMisses`). Neither path crosses a storage virtual. The abstract-base
design's "virtual cost is cold-path" claim is empirically supported.

## 6. Cross-component locking (combined-mutex invariant)

The single `storeMutex_` on `TraceStorage` protects:

- **Storage state:** SQLite connection, dedup maps (`resultByHash`,
  `depKeySetByHash`, `traceByFullHash`), pending buffers, ID counters.
- **Verifier state:** session-scoped caches (`traceHeaderCache_`,
  `traceFullCache_`, `currentNodeIndex_`, `resultPayloadCache_`,
  `depKeySetCache_`, `deferredTraceBlobs_`), `VerificationSession`
  (L1 cache + subsumption facts), `ParseCaches`.

This is why `FileStrandGate::ifPassed(ea, f)` can mint
`Proof<FileStrandTag>` from `ea` alone: the ParseCaches member lives on
`VerificationSession` (inside Verifier) and rides the same mutex. Every
mutating Verifier method takes `const ExclusiveTraceStorageAccess &`
(or is reachable only inside a `withExclusiveAccess` continuation). No
verifier-owned field has its own mutex.

**Non-recursive mutex safety argument.** Today's `recursive_mutex` is
load-bearing because private helpers (`ensureTraceHeader`,
`lookupCurrentNode`, `loadResultPayload` at trace-store.cc:192, :552,
:588) call `std::lock_guard` internally. Enumerated against the code,
every caller of these helpers already holds the mutex via `ea`:

| Helper | Callers (all inside `withExclusiveAccess`) |
|---|---|
| `lookupCurrentNode` | `attrExists`, `getCurrentTraceHash`, `TraceStore::verify`, `resolveTraceContextHash`, `verifyAttrImpl`, `queryEvalInfoExclusive` |
| `ensureTraceHeader` | `getCurrentTraceHash`, `tryStructuralVariantRecovery`, `patchTraceHashInMemory`, `queryEvalInfoExclusive`, `loadFullTrace` (cache population) |
| `loadResultPayload` via `decodeCachedResult(bs,…)` | `verifyAttrImpl`, `tryDirectHashRecovery`, `TraceStore::verify` sync path, `queryEvalInfoExclusive` |

All hold ea. The internal `lock_guard` calls are redundant. After §4's
migration, private helpers take `ea` directly, and the
`std::recursive_mutex → std::mutex` switch becomes safe. The
`thread_local` re-entrancy detector (trace-store-lifecycle.cc:52-80)
still fires on accidental nesting: debug builds assert, release builds
deadlock (strictly better than today's silent recursion masking the
bug).

**Coroutine discipline.** `verifyAttrImpl` (awaitable) reacquires `ea`
between coroutine suspensions: release across `co_await`, re-enter
`withExclusiveAccess` on resume.

**Exception.** `Verifier::prefetchPool_` is `StrandLocal<PrefetchPool,
VerificationAccessTag>`. Access is gated by
`Certifier<VerificationAccessTag>::withProof`, not by `ea`. Safe
because single-threaded eval ensures only one `verifyAttrImpl` runs at
a time.

## 7. Shutdown ordering

```cpp
~TraceBackend() {
    // 1. Stop blocking pool first. In-flight blocking threads finish
    //    their work, post timer cancellations to ioc_, then exit.
    blockingPool_.stop();
    // 2. Drain ioc_ — workers process the timer cancellations,
    //    resuming any suspended coroutines so they unwind cleanly.
    //    Release the work guard so run() returns. Stop + join ioc workers.
    iocWorkers_.shutdown();
    // 3. All threads stopped. Destroy recorder_ (refs only) then
    //    verifier_ (releases prefetch pool + session state + cache refs).
    recorder_.reset();
    verifier_.reset();
    // 4. storage_ destroyed last by member destructor. Verifier held
    //    TraceStorage& as raw ref; storage_ outlives verifier_ because
    //    storage_ is declared first, destructed in reverse order.
}
```

Declaration order (from §2.4): `storage_` first, then `ioc_`,
`scheduler_`, `workGuard_`, `iocWorkers_`, `blockingPool_`, `verifier_`,
`recorder_`, `sessionBound_`. Reverse destruction order mostly works
without the explicit body, but the explicit sequence is belt-and-braces:
it guarantees `blockingPool_.stop()` precedes `verifier_.reset()` so
verifier's coroBlock callbacks never fire after pool shutdown.

## 11. Risks and mitigations

- **Nested `withExclusiveAccess` re-entrancy.** Debug-build
  `thread_local` re-entry detector keys on `TraceStorage *`. Same
  invariant; no regression. With `std::mutex`, re-entry deadlocks in
  release builds (previously silently recursed — strictly better).
- **Cross-store nesting** (one thread holding `ea` on store A while
  acquiring `ea` on store B). Not forbidden; same risk as today.
- **`TraceSessionFactory` in `eval-environment/authority.cc`** calls
  `session->releaseBackend()` when the semantic digest changes.
  `releaseBackend` becomes `runtime_->flush() + runtime_.reset()`.
  Same semantics.
- **Runtime-root mount registration** still goes through the two-call
  `TraceSession::registerRuntimeRootMount` +
  `runtime_->recordRuntimeRoot`. A single-call atomic
  `publishRuntimeRoot` is out of scope.
- **Performance (benchmark-verified).** No algorithmic change. Per-verify
  delta is one virtual dispatch on each `TraceStorage::insertX / lookupX
  / publishX` call, which is not in the inner SV loop (the inner loop
  goes through `Verifier::resolveCurrentDepHash` → `resolveDepHash`
  free fn; the storage boundary is crossed only on the cold path of
  each *trace*, not each *dep*). Hot-mode measurements confirm storage
  calls are ~1% of verify time (506 ms loadTrace vs 42.5s verify total
  across 100 commits). Benchmark parity must be within noise.
- **I/O latent-optimization unlocked.** `loadTraceHeader` virtual is a
  header-only query; avoids loading the ~661 KB BLOB payload to extract
  72 B of scalar header. Not a requirement — today's `traceHeaderCache_`
  hides the miss cost within a session — but the rewrite captures this
  for free. Benchmark source: hot/0 DB, avg `keys_blob` = 49 KB, avg
  `values_blob` = 612 KB across 93 traces.
- **Build-time regression for `trace-session.hh` consumers.** The new
  `trace-backend.hh` transitively includes `verifier.hh` +
  `trace-storage.hh`. Today's `trace-session.hh` transitively exposes
  heavier types through `trace-store.hh` + 28 SQLiteStmts, so the new
  shape should be no worse. Budget: if the post-rearch full-build time
  regresses > 5% vs baseline, reopen and pImpl `runtime_` in
  `TraceSession`. Measurement deferred to commit time.
- **`TraceSession::BackendParams` fusion preserved.** Today's
  `trace-session.hh:54` defines a nested `BackendParams{fingerprint,
  sessionConfig}` that the TraceSession ctor takes as
  `std::optional<BackendParams>` (line 105). This fusion is the OR-8
  closure — `useCache=Some, sessionConfig=None` is now
  unrepresentable at construction time. The rewrite must preserve
  `BackendParams` as a nested type; the only change is that
  `setSessionConfig` is now a private method on `TraceBackend` called
  from `TraceSession`'s ctor instead of on `StoreTraceBackend`. No
  change to `BackendParams` shape or invariant.
- **Hash-domain preimage regression.** Pre-commit check:
  `grep -rhoP 'hash_domain::\w+' src/libexpr/ src/libflake/ | sort -u`
  must list the same 25 tags before and after the rename. The
  `hash_domain::NewDomain` mention in `canonical-hash.hh:138` is a
  doc-comment placeholder (it's the example name a new-domain author
  would use) and does not correspond to a real tag — safe to filter
  out with `grep -v NewDomain` if a difference surfaces. A
  namespace-mangling rename could silently change a preimage.

## Appendix A: accessor-identity hazard class

Separate from the rearchitecture above. Records a hazard class —
discovered after the accessor-preservation fixes landed in
`ExprConcatStrings::eval` and `EvalState::storePath` — that affects
`libexpr` path construction broadly and has implications for any future
storage- or verification-side design that interacts with path values.

### A.1 Symptom and scope

`SourceAccessor::operator==` compares by a monotonic `number` field
assigned at construction (`src/libutil/source-accessor.cc`,
`nextNumber` atomic). Two accessor instances wrapping the same
underlying filesystem location are never equal. When Nix path values
built through different accessors meet at a structural comparison site
— notably nixpkgs' `lib.fileset.toSource`, `lib.path.hasPrefix`, and
the fileset library's internal `splitRoot` uses — they compare unequal,
producing user-visible "Filesystem roots are not the same" errors even
when both paths resolve to the same on-disk store object.

The hazard surfaces whenever a runtime path construction drops or
replaces the accessor of a source path. Three manifestations:

- **Fixed.** `ExprConcatStrings::eval` for the `firstType == nPath`
  branch rebuilt the result via `state.rootPath(CanonPath(resultStr))`
  which wraps in `state.rootFS` regardless of the source path's
  accessor. Committed fix: capture `firstPathAccessor` on the first
  iteration and use it at construction.
- **Fixed.** `EvalState::storePath` returned `{rootFS, canonPath}`
  while flake machinery (`makeMountedStorePath`) uses `{storeFS,
  canonPath}` for the same store objects. Committed fix: route
  `storePath` through `storeFS`.
- **Latent (OR-11).**
  `src/libexpr/eval-trace/cache/materialize.cc`'s `path_t` restoration
  uses `st.rootPath(CanonPath(p->path))`. The original accessor was
  never persisted in `path_t`'s encoding, so the restored path carries
  `rootFS` unconditionally. Not demonstrably reachable in current
  user-visible behavior; requires a schema change or a
  `SemanticRegistry`-based lookup to fix. See OR-11 in
  `src/libexpr/eval-trace/CLAUDE.md` and DEF-9 in
  `src/libexpr-tests/eval-trace/CLAUDE.md`.

### A.2 Root cause — equality-policy mismatch

The evaluator treats a `SourcePath` as `(accessor_handle, canon_path)`
and compares by handle-identity. Nixpkgs-lib code treats paths as
*filesystem objects* and expects two values at the same on-disk
location to compare equal, regardless of which code path constructed
them. These two semantics are not reconciled in the codebase. Every
path-producing site has to propagate the accessor correctly, and any
site that "rebuilds from a string" via `state.rootPath` /
`state.storePath` silently becomes an accessor-identity hazard.

### A.3 Type-level prevention options

**Option A (lowest effort; narrow).** Replace `state.rootPath(...)`
call sites that rebuild an accessor-less path with explicit capture +
construction idioms like the `ExprConcatStrings` fix. Purely a
code-convention change; no new types. Does not prevent future
regressions introduced by new call sites.

**Option B (medium effort; localized).** `AccessorForwardingScope` —
an opaque capability following the sealed-construction pattern of
eval-trace's `OriginScope<CurrentTrace>` / `OriginDep<Origin>`:

```cpp
class AccessorForwardingScope {
    SourceAccessor * source_;
    explicit AccessorForwardingScope(SourceAccessor * s) : source_(s) {}
    friend class AccessorForwardingScopeFactory;  // sealed
public:
    Value mkForwardedPath(EvalMemory & mem, CanonPath newPath) const;
};
```

Private ctor; minting factory is friended and lives in a single TU.
Every path-producing site derived from an existing source path routes
through this scope. Sites that explicitly want `rootFS`-provenance
call `state.rootPath` directly and are readable as such.

**Option C (high effort; codebase-wide).** Make accessor-origin
nominal via `Tagged<Tag, SourceAccessor *>`. Phantom tags
(`StoreFSOrigin`, `RootFSOrigin`, potentially `MountedOrigin`) make
`SourcePath<Tag>` parametric on its origin. Path-producing operations
require matching tags or explicit type-checked coercions. Structurally
prevents the invariant violation but pervasive: every `SourcePath`
declaration, every `mkPath`-like function, every `Value::path()`
accessor needs to know or preserve the tag. Comparable in scale to the
`CurrentTraceDep`/`CandidateDep` distinction already shipped.

**Option D (long-term; semantic change).** Revisit
`SourceAccessor::operator==` so two handles addressing the same
filesystem location compare equal. Natural shapes: content-address
accessors so `getFSAccessor(storePath)` dedups on construction and
returns the same instance for the same store path; or introduce a
canonical identity separate from `number` — e.g., a `FilesystemRoot`
opaque handle that `MountedSourceAccessor` and its sub-accessors
share.

### A.4 Recommendation

Defer Option B and Option D until a new accessor-identity bug is
reported. When one is, land Option B (`AccessorForwardingScope`) as
the immediate mechanism and leave Option D as a standing design
question. Option D is the root-cause fix but requires auditing every
`SourceAccessor::operator==` caller (map keys, cache indexing, etc.)
to confirm they really want filesystem-identity rather than
handle-identity — an effort spanning libexpr, libutil, and potentially
nixpkgs-lib expectations.
